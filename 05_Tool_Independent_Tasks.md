# 工具獨立任務認領清單

這份文件詳細列出了在「互相不參考、各自獨立」的情況下，您所擁有的各個工具能夠獨立認領的開發任務。這有助於您在規格驅動開發（SDD）過程中，明確知道何時該呼叫哪個工具來完成特定工作。

## 1. ChatPRD (產品需求文件生成)

ChatPRD 專注於需求定義與規格草擬。當您有一個初步的靈感，例如想要建立一個符合 ASPICE 流程的文件管理系統時，ChatPRD 能夠獨立將這段非結構化的想法轉化為包含使用者故事、功能列表、非功能性需求和驗收標準的正式產品需求文件（PRD）。此外，針對特定功能如「文件相依性更新」，ChatPRD 可以獨立列出所有可能的邊界條件（例如網路斷線、權限不足或循環相依）及其對應的處理邏輯。在開發初期，它也能根據已定義的功能，獨立生成對應的測試案例草稿，供後續的測試驅動開發（TDD）使用。

| 獨立認領任務 | 執行方式 |
| :--- | :--- |
| 將靈感轉化為結構化 PRD | 輸入非結構化想法，生成包含使用者故事與驗收標準的正式 PRD。 |
| 定義邊界條件與例外處理 | 針對特定功能，列出所有可能的邊界條件及對應的處理邏輯。 |
| 生成測試案例草稿 | 根據已定義的功能，獨立生成對應的測試案例供 TDD 使用。 |

## 2. Warp (AI 終端機)

Warp 負責環境管理與終端機操作。在版本控制方面，您可以使用自然語言指示 Warp 執行 Git 操作，例如要求它將所有修改提交並推送到新的分支，Warp 會自動生成並執行對應的 Git 指令。當遇到開發環境配置問題或缺少依賴套件時，您可以直接在 Warp 中詢問解決方案，它會提供並執行如 `npm install` 或 `pip install` 等安裝指令。此外，當程式運行報錯或測試失敗時，Warp 的 AI 功能可以獨立分析錯誤日誌，並提供具體的修復建議。

| 獨立認領任務 | 執行方式 |
| :--- | :--- |
| Git 版本控制操作 | 使用自然語言指示，自動生成並執行對應的 Git 指令。 |
| 開發環境建置與依賴安裝 | 詢問環境配置問題，提供並執行依賴套件安裝指令。 |
| 終端機錯誤排解 | 分析程式運行報錯或測試失敗的日誌，並獲取修復建議。 |

## 3. Magic Patterns (AI 原型設計)

Magic Patterns 專注於前端介面與互動設計。您可以透過輸入文字描述，例如「一個包含文件列表、狀態標籤和操作按鈕的資料表」，讓 Magic Patterns 獨立生成對應的 React 或 Tailwind CSS 代碼。它也能根據您描述的使用者操作流程（例如點擊文件後彈出側邊欄顯示相依性），生成包含互動邏輯的前端原型。這使得團隊在不寫任何後端代碼的情況下，能夠快速生成多個版本的 UI 介面，以驗證設計概念並進行討論確認。

| 獨立認領任務 | 執行方式 |
| :--- | :--- |
| 生成 UI 元件代碼 | 輸入文字描述，獨立生成對應的 React/Tailwind CSS 代碼。 |
| 設計互動式流程 | 描述使用者操作流程，生成包含互動邏輯的前端原型。 |
| 快速驗證設計概念 | 快速生成多個版本的 UI 介面，供團隊討論和確認設計概念。 |

## 4. Mobbin (UI/UX 設計參考)

Mobbin 是尋找設計靈感與進行競品分析的利器。您可以在 Mobbin 中搜尋特定關鍵字，如「文件管理」或「版本控制」，獨立查看業界頂尖產品（如 Notion, Linear, GitHub）是如何設計這些介面的。它允許您收集特定 UI 模式（例如複雜表單、多層級導航或狀態指示器）的截圖，作為後續生成代碼的參考。透過將收集到的優秀設計整理成情緒板（Moodboard），Mobbin 能幫助您確立產品的視覺風格和互動基準。

| 獨立認領任務 | 執行方式 |
| :--- | :--- |
| 尋找特定功能的最佳實踐 | 搜尋關鍵字，查看業界頂尖產品的介面設計。 |
| 收集 UI 模式（Patterns） | 獨立收集特定 UI 模式的截圖，作為設計與開發的參考。 |
| 建立設計情緒板（Moodboard） | 將收集到的設計整理成情緒板，確立視覺風格和互動基準。 |

## 5. Factory (軟體開發自動化)

Factory 負責代碼審查與自動化開發任務。當您在 GitHub 提交 Pull Request 時，Factory 能夠獨立進行代碼審查，檢查代碼是否符合編碼規範，並揪出潛在的 Bug 或安全漏洞。在 CI/CD 流程中，Factory 會獨立運行單元測試和整合測試，確保新加入的代碼不會破壞現有功能。此外，根據已定義好的資料模型，Factory 可以獨立生成基礎的 CRUD API 路由、資料庫遷移腳本或前端 API 客戶端代碼，大幅減少撰寫常規代碼（Boilerplate）的時間。

| 獨立認領任務 | 執行方式 |
| :--- | :--- |
| 自動化代碼審查（Code Review） | 在提交 PR 時，獨立檢查編碼規範、潛在 Bug 或安全漏洞。 |
| 自動化測試執行 | 在 CI/CD 流程中獨立運行單元測試和整合測試。 |
| 生成常規代碼（Boilerplate） | 根據資料模型，獨立生成基礎的 API 路由、遷移腳本或客戶端代碼。 |

## 6. Railway (雲端部署)

Railway 專注於基礎設施與部署管理。您可以在 Railway 平台上獨立點擊建立 PostgreSQL 資料庫實例，並獲取連接字串供本地開發或雲端應用使用。透過將 Railway 連結到您的 GitHub 倉庫，您可以設定當特定分支（如 `main`）有更新時，Railway 獨立進行自動化構建和部署。更強大的是，當有新的 Pull Request 產生時，Railway 能夠獨立為該 PR 建立一個臨時的預覽環境（Preview Environments），方便團隊進行測試和審查。

| 獨立認領任務 | 執行方式 |
| :--- | :--- |
| 資料庫建置與管理 | 在平台上建立資料庫實例，獲取連接字串供開發使用。 |
| 自動化部署（CI/CD） | 連結 GitHub 倉庫，設定分支更新時自動進行構建和部署。 |
| 預覽環境生成 | 為新的 PR 獨立建立臨時的預覽環境，供測試和審查。 |

## 7. PostHog (產品分析)

PostHog 負責數據收集與使用者行為分析。透過在前端代碼中埋入 PostHog SDK，系統能夠獨立收集使用者的點擊、頁面瀏覽和操作路徑。您可以在 PostHog 後台獨立建立儀表板，分析特定功能（例如「相依性檢查」）的使用頻率和留存率。此外，開啟會話錄影（Session Recording）功能後，PostHog 會獨立記錄使用者在系統中的實際操作過程，這對於發現 UX 問題或重現難以捉摸的 Bug 非常有幫助。

| 獨立認領任務 | 執行方式 |
| :--- | :--- |
| 使用者行為追蹤 | 埋入 SDK，獨立收集使用者的點擊、頁面瀏覽和操作路徑。 |
| 功能使用率分析 | 建立儀表板，分析特定功能的使用頻率和留存率。 |
| 會話錄影（Session Recording） | 記錄使用者實際操作過程，用於發現 UX 問題或重現 Bug。 |

## 8. Manus (通用 AI 代理)

Manus 擅長處理複雜分析與跨領域整合任務。它能夠獨立閱讀和分析龐大的 PDF 或 Markdown 文件（例如 ASPICE 規範），並從中提取關鍵規則和屬性。根據您的需求，Manus 可以獨立設計複雜的系統架構（如多 Agent 協作模型）和資料庫 Schema（如 EAV 模型）。最後，Manus 也能獨立撰寫詳細的技術規格書、API 文件、操作手冊和開發流程指南，確保專案的知識傳承與文件完整性。

| 獨立認領任務 | 執行方式 |
| :--- | :--- |
| 深度文件分析與知識提取 | 獨立閱讀龐大文件，提取關鍵規則和屬性。 |
| 系統架構與資料模型設計 | 根據需求獨立設計複雜的系統架構和資料庫 Schema。 |
| 技術文件與操作指南撰寫 | 獨立撰寫技術規格書、API 文件、操作手冊和開發流程指南。 |
